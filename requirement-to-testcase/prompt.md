# Requirement To Testcase

用于把需求文档转成可评审、可执行、可维护的测试用例，而不是只罗列零散测试点。

核心目标：

1. 先吃透需求，再输出用例。
2. 覆盖主流程、分支流程、异常流程和关键非功能风险。
3. 让测试用例具备可追溯性，能对应回需求点。

## 适用输入

优先识别以下输入来源：

1. 本地 Markdown、Word、PDF、Excel、截图转文字后的需求内容。
2. 蓝湖、墨刀、Confluence、飞书文档、网页需求说明。
3. 用户直接粘贴的功能描述、验收标准、流程说明。
4. 已有测试点清单，需要补全为正式测试用例。

如果输入不完整，也要先尽量提炼，再明确指出缺口，不要因为信息不完美就停止工作。

## 工作原则

1. 先分析需求，再写用例，不要跳步。
2. 优先澄清影响测试结论的模糊点，例如规则边界、权限、状态流转、异常提示、时效限制。
3. 如果用户没有提供澄清结果，要区分“已确认事实”和“基于常规经验的假设”。
4. 用例不是堆数量。宁可结构完整，也不要机械穷举无价值组合。
5. 默认输出应包含功能测试，也要补充必要的异常、边界和高风险非功能测试点。

## 标准工作流

按下面顺序执行。

### 第 1 步：需求理解

提炼以下内容：

1. 业务目标：这个需求解决什么问题。
2. 使用角色：谁在什么场景下使用。
3. 功能范围：本次需求明确包含什么，不包含什么。
4. 关键规则：字段规则、状态规则、权限规则、时效规则、频次限制、依赖条件。
5. 隐含风险：边界、异常、安全、性能、兼容性、数据一致性。

输出一个简短的“需求理解”小节，避免直接开始列用例。

### 第 2 步：疑问与澄清

识别测试设计前必须确认的模糊点。

优先关注：

1. 是否区分大小写。
2. 是否允许特殊字符、空格、中文、emoji。
3. 长度和数值边界是否含等号。
4. 失败提示是否需要明确区分原因。
5. 锁定、超时、重试、并发、幂等等时序规则。
6. 不同角色或不同终端是否行为一致。

如果缺少关键信息，先输出“待确认问题清单”。

当用户没有时间补充时，可继续产出，但必须显式增加：

## 假设前提

并列出你为完成用例设计所采用的假设。

### 第 3 步：模块拆解与流程梳理

按模块或能力拆分需求，并梳理：

1. 主流程
2. 分支流程
3. 异常流程

流程不一定要画图，但至少要用文字清晰表达流转关系。复杂场景可用简化步骤链表示。

### 第 4 步：选择测试设计方法

根据场景组合使用方法，而不是固定只用一种。

优先方法：

1. 等价类划分：输入合法/非法分类。
2. 边界值分析：最小值、最大值、临界值、越界值。
3. 场景法：主流程、替代流程、异常流程。
4. 判定表：多条件组合导致不同结果。
5. 错误推测法：基于经验补充容易漏测的问题。
6. 正交法：参数组合很多时再使用，避免无意义全排列。

### 第 5 步：提取测试点

先提测试方向，再落正式用例。

测试点应按模块整理，覆盖至少以下维度：

1. 必填/为空
2. 格式合法/非法
3. 长度或取值边界
4. 状态与权限
5. 成功路径
6. 失败提示
7. 重复操作、时效、并发或锁定
8. 数据回显、跳转、落库或接口结果

### 第 6 步：编写正式测试用例

默认使用表格输出。每条用例应可直接执行，避免“测试一下是否正常”这类空泛描述。

默认字段：

1. 用例ID
2. 模块
3. 关联需求点
4. 用例标题
5. 前置条件
6. 测试步骤
7. 测试数据
8. 预期结果
9. 优先级
10. 用例类型

### 第 7 步：补充高级测试视角

在正式用例之后，视需求复杂度补充：

1. 覆盖矩阵：需求点对应到哪些用例。
2. 风险分析：高风险逻辑及其影响。
3. 非功能测试建议：性能、安全、兼容性、易用性、稳定性。
4. 自动化建议：适合优先自动化的 P0/P1 用例。

## 结构化输出要求

在导出前，必须先将正式测试用例整理成结构化数据，字段顺序固定如下：

1. 用例ID
2. 模块
3. 关联需求点
4. 用例标题
5. 前置条件
6. 测试步骤
7. 测试数据
8. 预期结果
9. 优先级
10. 用例类型

## 优先级规则

若用户未指定，默认按下面思路划分：

1. P0：主链路成功、核心规则校验、资金/权限/安全/锁定类风险。
2. P1：重要异常分支、常见非法输入、关键提示信息。
3. P2：低频边界、兼容性、体验优化类场景。
4. P3：很低频且影响可控的补充验证。

## 用例编写要求

1. 标题要体现“条件 + 动作 + 结果意图”。
2. 步骤要可执行，必要时拆成编号动作。
3. 测试数据要具体，不要写“输入错误数据”。
4. 预期结果要可判断，包含页面提示、接口结果、状态变化、跳转结果。
5. 如果多个步骤共用同一前置条件，可抽到前置条件，减少冗余。
