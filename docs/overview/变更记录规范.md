# 变更记录规范

本文档用于约束后续手动维护或自动维护虾如何记录服务器变更。

## 1. 记录目标

每次对服务器运行目录做出真实改动后，都要在 `docs/overview/服务器变更日志.md` 追加一条记录。

这里记录的是“变更事件”，不是字段清单。

## 2. 何时必须记录

以下情况必须追加日志：

- 修改运行目录下的脚本、配置、模块说明、agent 配置、文档
- 调整路由逻辑、模块收敛关系、权限映射、维护规则
- 修复线上配置错误、软链接错误、编码错误、脚本入口错误
- 新增或删除服务器侧文档、日志规则、维护约定

以下情况不要追加日志：

- 纯只读检查
- 没有落盘的临时排查
- 重复描述同一轮未新增信息的操作

## 3. 每条记录最少字段

每条记录至少要包含：

- 时间
- 来源
- 位置
- 变更
- 原因
- 验证

推荐模板：

```md
## 2026-04-03 02:40 CST

- 来源：Codex / 手动维护
- 位置：`docs/overview/模块收敛计划.md`、`modules/task-close-flow/SKILL.md`
- 变更：新增模块收敛计划，并把 task-close-flow 标记为 deprecated / 待收敛
- 原因：避免重复扩功能，明确后续统一向主模块扩展
- 验证：已抽查文档内容；待收敛模块中已出现目标收敛模块说明
- 备注：未删除任何目录或脚本
```

## 4. 自动记录虾的约束

如果后续创建自动记录服务器变更的维护虾，应遵守：

- 只在检测到真实文件改动后追加日志
- 不要凭空生成“看起来合理”的变更记录
- 一次改动尽量合并成一条高质量记录，不要拆成很多碎条目
- 日志内容要基于真实 diff、真实文件路径和真实验证结果
- 如果无法确认验证结果，要明确写“未验证”，不要伪造成功

## 5. 当前建议

- 对于已经形成 git 提交的功能改动，优先执行 `python3 scripts/maintenance/record_server_change_after_commit.py HEAD` 自动补记日志。
- 自动补记时，“位置”默认只写关键文件，完整文件清单以 `git show --stat <commit>` 为准。
- 推荐顺序为：`git commit` -> `python3 scripts/maintenance/record_server_change_after_commit.py HEAD` -> `git push`。
- 如果脚本返回 `SERVER_CHANGE_ALREADY_RECORDED`，说明当前提交已留痕，不必重复追加。
- 如果后续为了保留日志再单独提交 `docs/overview/服务器变更日志.md`，自动记录脚本应跳过该类仅日志提交，避免递归追加。

- 服务器主文档继续放在 `docs/overview/`
- 本地文档只保留为入口或镜像
- 后续所有服务器事实更新，优先更新服务器文档，再同步本地摘要
