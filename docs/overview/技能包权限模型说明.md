# openclaw-zentao-pack 技能包权限模型说明

本文档用于为 openclaw-zentao-pack 设计一套可落地的权限模型，目标是实现：

- 用户在企业微信中发起请求时，能够识别其对应的禅道身份
- 用户只能执行其角色允许范围内的禅道功能
- 查询、创建、状态流转、用户同步等能力都能纳入统一鉴权体系
- 后续可扩展为模块级、脚本级、动作级、项目级权限控制

## 1. 背景与问题

当前技能包已经具备完整的业务能力，但权限控制还没有形成统一模型。

### 1.1 当前已有能力

当前代码里已经具备以下基础：

- 支持通过 userid 识别当前企业微信用户
- 支持通过 findUserByUserid 匹配禅道用户
- 支持通过 syncWecomUser 把企业微信用户同步到禅道
- 支持在配置中设置 default_role、default_group、group_rules.by_department、group_rules.by_department_leader、group_rules.by_position_keyword
- 大多数脚本已经统一支持 --userid

### 1.2 当前缺口

但从权限控制角度看，当前仍存在明显缺口：

- 没有统一的 authorize 鉴权入口
- 没有把模块、脚本、动作抽象成权限点
- 没有把禅道角色或组映射成技能包可执行动作集合
- 没有项目级或执行级范围限制
- 没有统一的高风险操作控制
- 没有形成角色、动作、资源三层模型

也就是说，当前更像是能识别是谁，但还不是能限制他做什么。

## 2. 权限模型目标

建议本技能包采用以下目标模型。

### 2.1 总体目标

把权限控制统一收敛为：

- 身份识别
- 角色映射
- 动作鉴权
- 资源范围判断
- 高风险操作加固
- 审计记录

### 2.2 最终控制公式

建议统一按下面的逻辑判断是否允许执行：

允许执行 = 身份已识别 且 角色有该动作权限 且 资源范围允许 且 高风险校验通过

## 3. 推荐的权限结构

建议采用四层模型。

### 3.1 用户身份层

负责回答：当前请求到底是谁发起的。

输入来源通常是：

- 企业微信 userid
- 回调 payload 中的 userid 或 userId
- 显式传入的 --userid

识别流程建议为：

1. 从消息或回调中提取企业微信 userid
2. 在禅道中查找对应用户
3. 读取禅道账号、真实姓名、角色、组、部门等信息
4. 生成当前技能包内部的 actor 对象

建议统一抽象为：

- actor.userid
- actor.account
- actor.realname
- actor.group
- actor.role
- actor.source

### 3.2 角色层

负责回答：这个用户属于什么权限角色。

建议不要只依赖单一字段，而应做多源映射：

- 禅道用户组 group
- 禅道 role 字段
- 企业微信岗位关键词
- 企业微信部门归属
- 企业微信部门负责人身份

建议最终统一映射成技能包内部角色，例如：

- viewer
- pm
- dev
- qa
- release_manager
- project_manager
- admin

### 3.3 动作层

负责回答：这个角色能不能执行这个动作。

建议不要直接按脚本名做权限，而是抽象成动作权限点。

例如：

- product.read
- product.create
- product.module.create
- story.read
- story.create
- story.review
- story.close
- task.read
- task.create
- task.update
- task.close
- bug.read
- bug.create
- bug.assign
- bug.resolve
- bug.close
- release.read
- release.create
- release.update
- release.link
- testcase.read
- testcase.create
- testtask.read
- testtask.create
- testtask.update
- testtask.run
- delivery.read
- acceptance.read
- user.sync
- callback.handle
- team.read
- team.manage

### 3.4 资源范围层

负责回答：这个动作是不是允许作用于当前资源。

例如：

- 只能查看自己参与的项目
- 只能查看自己所在执行的任务
- 只能修改分配给自己的任务
- 只能处理自己负责的 Bug
- 只能操作本部门或本项目的数据

推荐资源范围至少支持：

- global：全局可见
- self：仅本人相关
- project_member：仅参与项目
- execution_member：仅参与执行
- department_scope：仅本部门范围
- assigned_scope：仅分配给自己

## 4. 推荐的角色模型

建议先从一套简单但能落地的角色模型开始。

### 4.1 推荐角色列表

- viewer：只读查看者
- pm：产品经理或需求负责人
- dev：研发
- qa：测试
- release_manager：发布负责人
- project_manager：项目负责人
- admin：管理员

### 4.2 推荐角色能力示例

viewer：只允许基础查询，不允许创建、更新、同步。

pm：可执行产品、模块、需求、发布相关查询与创建，适合产品经理。

dev：可执行任务、Bug 修复相关查询与动作，适合研发。

qa：可执行测试用例、测试单、测试执行、Bug 提交与回归相关动作。

release_manager：可执行发布、交付、验收相关查询与发布动作。

project_manager：可执行项目、执行、团队、收口类协调动作。

admin：可执行全部查询与动作。

## 5. 动作与脚本映射方式

后续要真正落地鉴权，建议建立一份统一映射表。

### 5.1 示例映射思想

把脚本映射到 action：

- scripts/create_story.ts 对应 story.create
- scripts/review_story.ts 对应 story.review
- scripts/update_story_status.ts 对应 story.close 或 story.update
- scripts/create_task.ts 对应 task.create
- scripts/update_task_status.ts 对应 task.update
- scripts/create_bug.ts 对应 bug.create
- scripts/assign_bug.ts 对应 bug.assign
- scripts/update_bug_status.ts 对应 bug.resolve 或 bug.close
- scripts/create_release.ts 对应 release.create
- scripts/update_release_status.ts 对应 release.update
- scripts/sync_user.ts 对应 user.sync

### 5.2 模块与 action 的关系

模块可以映射为一组 action 的集合。

例如：

- story-review-flow 对应 story.create、story.review、story.read
- task-status-flow 对应 task.update
- bug-assign-flow 对应 bug.assign、bug.read
- test-execution-flow 对应 testtask.run、testtask.read、testcase.read

这样就能进一步实现：

- 角色到 action
- action 到脚本
- 模块到 action 集

## 6. 当前代码里的最佳落地点

从当前代码结构看，最适合加权限模型的位置有两层。

### 6.1 第一层：脚本入口层

所有根级业务脚本入口在创建 ZentaoClient({ userid }) 后，都应先执行统一鉴权。

例如：

- create_story.ts
- create_task.ts
- create_bug.ts
- create_release.ts
- update_task_status.ts
- update_bug_status.ts
- update_release_status.ts
- sync_user.ts

建议统一加一个权限入口，例如：authorizeAction({ userid, action, resource })。

### 6.2 第二层：共享客户端层

在 scripts/shared/zentao_client.ts 中，对关键动作再次做兜底校验，防止有人绕过入口直接调用内部方法。

这样可以形成：

- 外层脚本鉴权
- 内层客户端兜底鉴权

## 7. 推荐新增的权限文件

建议后续新增一个共享权限文件，例如 scripts/shared/permission.ts。

职责建议包括：

- 定义角色枚举
- 定义 action 枚举
- 定义角色与 action 映射表
- 定义资源范围判断函数
- 定义统一 authorizeAction 方法
- 定义高风险动作集合

## 8. 高风险动作建议

有些动作不应只做普通角色校验，还应增加一层风险控制。

建议列为高风险动作：

- 用户同步
- 发布状态变更
- 生命周期收口类关闭动作
- 批量同步组织用户
- 可能影响多人协作结果的状态流转

建议加固方式：

- 仅允许特定角色执行
- 要求显式确认参数
- 写审计日志
- 后续支持二次确认或审批口令

## 9. 审计与日志建议

权限模型落地后，建议统一记录以下信息：

- 谁发起的请求
- 对应企业微信 userid
- 对应禅道 account
- 对应角色
- 请求的 action
- 作用的资源编号
- 是否允许执行
- 拒绝原因
- 执行时间

## 10. 当前阶段推荐的最小落地方案

如果不想一开始做得太重，建议先落地一个最小版本。

### 10.1 第一步

先定义：

- 角色枚举
- action 枚举
- 脚本到 action 映射表

### 10.2 第二步

实现统一鉴权函数：

- 输入：userid、action、资源信息
- 输出：允许或拒绝，以及原因

### 10.3 第三步

把以下高频入口先接上鉴权：

- create_story.ts
- create_task.ts
- create_bug.ts
- create_release.ts
- update_task_status.ts
- update_bug_status.ts
- update_release_status.ts
- sync_user.ts

### 10.4 第四步

再逐步补查询范围限制：

- query_projects.ts
- query_execution_tasks.ts
- query_project_team.ts
- query_execution_team.ts
- get_my_tasks.ts

## 11. 与现有配置的关系

当前配置文件中已有：

- default_role
- default_group
- group_rules
- user_aliases

这些配置可以继续保留，但职责应重新明确。

### 11.1 当前配置适合做什么

- 用户同步默认值
- 企业微信用户到禅道组或角色的初始映射
- 部门与岗位到禅道组的同步规则

### 11.2 当前配置不够做什么

- 不能直接代替技能包动作鉴权
- 不能直接决定脚本是否允许执行
- 不能直接决定项目级资源范围

所以建议把它定义为身份与组织映射层配置，而不是最终权限控制层配置。

## 12. 文档结论

当前技能包已经具备了识别用户和执行业务动作的基础，但还缺一层把两者真正连接起来的权限网关。

最推荐的落地方向是：

1. 抽象 action 权限点
2. 把禅道 group 或 role 映射成技能包角色
3. 在脚本入口统一执行 authorize
4. 在共享客户端关键动作里做兜底校验
5. 增加资源范围控制与审计日志

只有这样，才能真正实现：

- 用户对应什么角色
- 就只能执行该角色范围内的禅道功能

## 13. 下一步建议

基于本文档，下一份最适合继续补的文档是：

- 角色与模块权限映射表

这样就可以把角色、模块、脚本、action 四层关系真正连起来。
