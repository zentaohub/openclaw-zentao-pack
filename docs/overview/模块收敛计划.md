# 模块收敛计划

本文档用于明确 `openclaw-zentao-pack` 中待收敛模块的去向，避免后续继续重复建设相同功能。

## 1. 收敛原则

- 不删除仍有用的脚本能力，但停止在重复模块上继续扩功能。
- 同一类业务能力只保留一个主扩展入口，避免多个模块重复承载相同职责。
- 收敛顺序优先是：先收敛文档和路由，再收敛模块说明，最后才考虑删目录。
- 如果一个旧模块仍被历史文档或角色映射引用，可先保留目录，但必须明确标记为 `deprecated / 待收敛`。

## 2. 收敛映射表

### 2.1 task-close-flow

- 状态：`deprecated / 待收敛`
- 功能现状：任务完成、关闭等动作已可由 `task-status-flow` 覆盖。
- 目标模块：`task-status-flow`
- 收敛规则：以后所有任务状态流转能力，只在 `task-status-flow` 扩展，不再在 `task-close-flow` 单独增加新逻辑。

### 2.2 bug-regression-flow

- 状态：`deprecated / 待收敛`
- 功能现状：回归验证场景同时涉及“待回归 Bug 查询”和“Bug 状态流转”。
- 目标模块：查询收敛到 `testing-bugflow`，状态更新收敛到 `bug-status-flow`
- 收敛规则：后续如果要增强回归验证，不再新增独立回归模块功能，而是在上述两个主模块中补查询或状态能力。

### 2.3 testtask-status-flow

- 状态：`deprecated / 待收敛`
- 功能现状：测试单生命周期能力存在，但当前未形成稳定的企微主路由模块入口。
- 目标模块：统一并入测试单主链路，即 `testtask-create-flow`、`test-execution-flow`、`test-exit-readiness-flow`
- 收敛规则：测试单的创建、执行、准出、状态演进以后视为一条主链路，不再单独孵化平行状态模块。

### 2.4 lifecycle-closure-flow

- 状态：`deprecated / 待收敛`
- 功能现状：和 `acceptance-closure` 都在做收口视图，存在明显重叠。
- 目标模块：`acceptance-closure`
- 收敛规则：后续所有收口视图、关闭准备度、阻塞项、验收总结等内容统一并到 `acceptance-closure`，不再分叉维护。

## 3. 当前主扩展入口

为了避免重复建设，当前建议把下面这些视为唯一主扩展入口：

- 任务状态相关：`task-status-flow`
- Bug 状态相关：`bug-status-flow`
- 测试视角查询：`testing-bugflow`
- 测试单主链路：`testtask-create-flow`、`test-execution-flow`、`test-exit-readiness-flow`
- 生命周期/验收收口：`acceptance-closure`

## 4. 执行建议

1. 新增需求前，先判断是否属于上述主扩展入口之一。
2. 如果属于，直接在主模块及其脚本上扩展，不再新增平行模块。
3. 如果后续确认旧模块再无独立价值，再删除对应目录，并同步更新：
   - `docs/overview/当前项目状态总览.md`
   - `docs/overview/技能包总体功能说明.md`
   - `docs/overview/模块与脚本映射清单.md`
   - `docs/overview/role-module-permissions.md`
