# 发送决策矩阵

## 1. 目标

这份文档用于明确：

- 什么角色在什么时间收到什么类型的消息
- 什么风险进入摘要
- 什么风险允许额外即时提醒
- 多角色用户如何合并

目的是把“是否发送、发送什么、发几条”固化为可执行规则，避免实现时各自理解不同。

## 2. 基本原则

- 同一用户同一时间槽只发 1 条摘要
- 风险默认进入 09:00 / 18:00 摘要
- 只有 P0 重大风险允许额外即时提醒
- 如果同一事项已在风险区展示，不再在普通待办区重复展开

## 3. 时间槽定义

| 时间槽 | 默认时间 | 目标 | 核心内容 |
| --- | --- | --- | --- |
| morning | 工作日 09:00 | 今日关注 | 待办、超期、阻塞、今日风险 |
| evening | 工作日 18:00 | 今日收口 | 今日完成、今日新增风险、未收口重点 |
| immediate | 非固定 | 重大风险即时提醒 | 仅 P0 事件 |

## 4. 角色摘要矩阵

| 角色 | 09:00 必带内容 | 18:00 必带内容 | 默认详情入口 |
| --- | --- | --- | --- |
| 产品 `pm` | 需求总量、高优需求、延期需求、今日重点需求 | 今日需求变更、今日关闭、延期需求、版本风险 | 我的需求 / 产品需求 |
| 研发 `dev` | 待办任务、超期任务、待处理 Bug、阻塞项 | 今日完成任务、今日解决 Bug、未收口超期项 | 我的任务 / 我的 Bug |
| 测试 `qa` | 待验证 Bug、待执行测试、回归 Bug、准出风险 | 今日执行情况、失败用例、阻塞用例、未收口风险 | 测试详情 / Bug |
| 管理 `manager` | 项目总量、延期项目、高风险执行、今日重点风险 | 今日推进、延期变化、准出状态、关闭准备度 | 项目进度 / 风险详情 |

## 5. 风险进入摘要矩阵

| 风险类型 | 产品 | 研发 | 测试 | 管理 | 09:00 | 18:00 | 允许即时提醒 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 项目延期 | 否 | 否 | 否 | 是 | 是 | 是 | 是 |
| 执行延期 | 可选 | 可选 | 可选 | 是 | 是 | 是 | 是 |
| 需求延期 | 是 | 否 | 否 | 是 | 是 | 是 | 否 |
| 普通任务延期 | 否 | 是 | 否 | 汇总 | 是 | 是 | 否 |
| 关键任务超期且 blocked | 否 | 是 | 否 | 是 | 是 | 是 | 是 |
| 普通 Bug 超期 | 否 | 是 | 是 | 汇总 | 是 | 是 | 否 |
| 高优 Bug 超期 | 可选 | 是 | 是 | 是 | 是 | 是 | 是 |
| 测试准出失败 | 否 | 否 | 是 | 是 | 是 | 是 | 是 |
| 关闭准备度不足 | 否 | 否 | 否 | 是 | 否 | 是 | 可选 |
| 发布风险 | 否 | 否 | 是 | 是 | 可选 | 是 | 可选 |

说明：

- `可选` 表示首期可不单独展开，只纳入汇总数字或管理摘要。
- `汇总` 表示进入管理视图中的风险计数，但不一定列入 Top3。

## 6. 重大风险即时提醒矩阵

| 风险事件 | 接收人 | 触发条件 | 再次提醒条件 |
| --- | --- | --- | --- |
| 项目首次延期 | 管理 | 首次从未延期变为延期 | 延期天数跨阈值，如 1 天 -> 3 天 -> 7 天 |
| 执行首次延期 | 管理、执行 owner | 首次进入延期 | 延期继续恶化 |
| 高优 Bug 超期 | 研发、测试、管理 | 超过高优阈值且未关闭 | 激活后再次超期，或超期天数跨阈值 |
| 关键任务 blocked 且超期 | 研发、管理 | 同时满足 blocked + overdue | 状态未改善且超期继续增加 |
| 测试准出硬阻塞 | 测试、管理 | 发布临近但仍不满足准出条件 | 阻塞项恶化或数量增加 |
| 关闭准备度由可关闭变为不可关闭 | 管理 | readiness 状态翻转 | 不建议重复提醒，等待状态改善或进一步恶化 |

## 7. 多角色合并矩阵

| 角色组合 | 默认合并方式 | 最大分区数 | 重点条目上限 |
| --- | --- | --- | --- |
| `dev + manager` | 我的待办 + 我的风险 | 2 | 4 |
| `pm + manager` | 我的需求 + 项目风险 | 2 | 4 |
| `qa + manager` | 我的测试 + 准出/交付风险 | 2 | 4 |
| `pm + dev` | 不推荐长期存在，按职责重定义 | 2 | 4 |

规则：

- 多角色合并后，总消息仍以“一屏读完”为目标。
- 合并时优先保留风险，不强求展示所有角色普通待办。

## 8. 发送抑制矩阵

| 场景 | 是否发送 | 说明 |
| --- | --- | --- |
| 没有任何待办，也没有任何风险 | 是 | 发一条“当前无高优风险”的保活摘要 |
| 只有普通待办，没有风险 | 是 | 发简版摘要 |
| 只有风险，没有普通待办 | 是 | 发风险优先摘要 |
| 重大风险刚发过，即将进入早/晚摘要 | 是 | 摘要照常发，但不重复单发同一风险 |
| 当天同一时间槽重跑任务 | 否 | 依赖去重键抑制重复发送 |

## 9. 重点项排序规则

同一条消息内部，重点项排序顺序建议固定为：

1. P0 重大风险
2. P1 重点风险
3. 今日必须处理的待办
4. 其他统计信息

同等级内排序建议：

1. 延期天数更多的优先
2. 优先级更高的优先
3. 更接近发布/截止时间的优先

## 10. 查看入口矩阵

| 角色 | 主入口 | 次入口 |
| --- | --- | --- |
| 产品 | 查看我的需求 | 查看产品需求 |
| 研发 | 查看我的任务 | 查看我的 Bug |
| 测试 | 查看测试详情 | 查看 Bug |
| 管理 | 查看项目进度 | 查看风险详情 |

要求：

- 每条摘要至少有 1 个入口
- 最多 2 个入口
- 不在一条消息里堆过多按钮或命令

## 11. 边界决策

### 11.1 一个人没有角色怎么办

- 默认不发
- 写日志
- 列为配置缺失项

### 11.2 一个人有角色但没有有效 scope 怎么办

- 个人角色优先走“我的任务 / 我的 Bug / 我的需求”
- 管理角色无 scope 时不建议发，避免口径失真

### 11.3 同一风险命中多个角色怎么办

- 多角色用户合并
- 不同用户各自收到与自己角色相关的视角
- 不要求所有角色文案完全一致
