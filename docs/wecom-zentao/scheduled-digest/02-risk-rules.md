# 延期与风险判定规则

## 1. 规则目标

本规则用于统一定义：

- 什么算延期
- 什么算阻塞
- 什么风险要进早晚摘要
- 什么风险允许额外即时提醒

## 2. 风险等级

## 2.1 P0 重大风险

允许额外即时提醒，同时保留在早晚摘要中。

包括：

- 项目/执行已延期
- 高优 Bug 超期未关闭
- 关键任务超期且 blocked
- 测试准出硬阻塞
- 关闭准备度关键条件不满足

## 2.2 P1 重点风险

只进入早晚摘要，不额外打扰。

包括：

- 普通任务延期
- 普通需求延期
- 中优 Bug 持续未收口
- 回归 Bug 数量上升

## 2.3 P2 普通提醒

用于总览数字，不单独占用重点条目。

包括：

- 待办数量变化
- 今日新增但未超期的普通事项

## 3. 项目 / 执行延期

命中条件：

- 当前日期晚于计划结束日期
- 状态不是完成、关闭、结束类状态

建议文案：

- `执行「结算改造」已延期 3 天`
- `项目「商城二期」距计划结束仅 1 天`

进入渠道：

- 管理早报
- 管理晚报
- 如首次进入延期状态，触发重大风险即时提醒

## 4. 任务延期

命中条件：

- `deadline` 已过
- 状态不属于 `done / closed / cancel / canceled`

升级为重大风险条件：

- 已超期
- 且状态为 `blocked`

建议文案：

- `任务 #231 已超期 2 天`
- `任务 #245 已超期且当前 blocked`

进入渠道：

- 研发早报
- 研发晚报
- 管理摘要中的团队风险汇总

## 5. Bug 超期

## 5.1 高优 Bug

推荐阈值：

- `P1` 或 `S1`：24 小时
- `P2` 或 `S2`：3 天

命中条件：

- Bug 未关闭
- 且创建或激活后已超过阈值

建议文案：

- `Bug #542 P1 已超期 2 天`
- `高优 Bug 2 个超期未处理`

进入渠道：

- 研发摘要
- 测试摘要
- 管理摘要
- 允许额外即时提醒

## 5.2 普通 Bug

命中条件：

- 未关闭
- 且已超过团队设定处理时限

进入渠道：

- 只进入摘要
- 默认不触发即时提醒

## 6. 需求延期

命中条件：

- 需求计划完成日期已过
- 状态未完成、未关闭

增强判断：

- 如果版本计划变化次数过多，也可标记为“计划波动风险”

建议文案：

- `需求 #1023 已延期 4 天`
- `版本 V3.2 有 2 项需求存在延期风险`

进入渠道：

- 产品摘要
- 管理摘要

## 7. 测试准出风险

命中条件满足任一项即可：

- 存在失败用例
- 存在阻塞用例
- 存在未执行用例
- 存在未关闭关联 Bug
- 测试单未完成

升级为重大风险条件：

- 发布临近且仍存在失败/阻塞/未关闭高优 Bug

建议文案：

- `测试准出未通过：失败用例 3，未关闭 Bug 2`
- `测试单未完成，仍有 5 个未执行用例`

进入渠道：

- 测试摘要
- 管理摘要
- 允许即时提醒

## 8. 关闭准备度风险

命中条件满足任一项即可：

- 未完成任务大于 0
- 未关闭 Bug 大于 0
- 活跃需求大于 0
- 没有有效发布记录

建议文案：

- `关闭条件未满足：未完成任务 4`
- `关闭准备度不足：未关闭 Bug 2`

进入渠道：

- 管理晚报
- 版本/发布负责人摘要

## 9. 风险展示优先级

同一条消息内，重点展示顺序建议固定为：

1. 项目 / 执行延期
2. 高优 Bug 超期
3. 关键任务延期 / blocked
4. 测试准出风险
5. 需求延期

如果风险超过 3 条：

- 只展示前三条
- 其余折叠为汇总数字

## 10. 风险去重原则

避免同一问题在一条消息里重复出现。

例如：

- 高优 Bug 已在“风险重点”列出后，不再在“普通待办”重复展开
- 任务既是超期又是 blocked，合并成一条描述

## 11. 风险状态变化触发规则

为避免重复打扰，建议只在以下状态变化时触发即时提醒：

- 首次进入延期
- 风险等级从 P1 升为 P0
- 已提醒后再次恶化，例如延期天数跨阈值

不建议在以下情况下重复提醒：

- 风险状态未变化
- 只是仍然存在但没有恶化
- 当天早晚摘要已覆盖且无新增升级
