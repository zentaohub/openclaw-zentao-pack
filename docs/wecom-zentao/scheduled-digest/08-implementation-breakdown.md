# 开发任务拆解

## 1. 目标

这份文档用于把方案拆成可排期、可分工、可验收的开发任务。

适用场景：

- 研发排期
- 任务拆分
- 联调前自查

## 2. 建议模块拆分

建议按以下 8 个工作包推进：

1. 配置加载与校验
2. 调度入口
3. 接收人解析
4. 数据采集与聚合
5. 风险判定引擎
6. 消息渲染
7. 发送与审计
8. 联调与验收

## 3. 工作包明细

## 3.1 配置加载与校验

目标：

- 支持读取 `scheduled-digest.json`
- 支持字段校验
- 启动时输出清晰错误

交付物：

- 配置结构定义
- 配置校验函数
- 示例配置文件

验收点：

- 非法角色会报错
- 重复 userid 会报错
- 缺少 zentaoAccount 会报错

## 3.2 调度入口

目标：

- 打通 morning / evening 两个统一入口
- 支持指定时间槽手动执行

交付物：

- `run-scheduled-digest --timeslot morning`
- `run-scheduled-digest --timeslot evening`
- 调度注册配置

验收点：

- 能手动触发
- 能被 cron 稳定调用
- 同一时间槽重跑不重复发送

## 3.3 接收人解析

目标：

- 从配置中生成本次待发送用户列表
- 处理用户开关、时间槽偏好、多角色合并

交付物：

- receiver resolver
- user -> roles -> scope 聚合逻辑

验收点：

- 同一用户多个角色只产出一份汇总任务
- 关闭 `receiveMorning` 后不进入早报列表

## 3.4 数据采集与聚合

目标：

- 复用现有查询脚本拉取角色数据
- 统一转为 digest collector 可消费的结构

交付物：

- `collectPmDigestData`
- `collectDevDigestData`
- `collectQaDigestData`
- `collectManagerDigestData`

验收点：

- 查询失败只影响当前用户
- 原始数据能转为统一 summary 结构

## 3.5 风险判定引擎

目标：

- 根据统一规则生成 P0 / P1 / P2 风险
- 做优先级排序和去重

交付物：

- `evaluateRisks(summary, riskRules)`
- 风险去重与升级判断

验收点：

- blocked 且超期任务升级为 P0
- 高优 Bug 超过阈值进入 P0
- 风险排序符合预期

## 3.6 消息渲染

目标：

- 按角色和时间槽渲染短消息
- 处理多角色合并和长度控制

交付物：

- `renderPmMorning`
- `renderDevEvening`
- `renderMergedDigest`
- 标题截断和 `topN` 控制

验收点：

- 单条消息重点不超过 3 条
- 合并消息分区不超过 2 个
- 没有长段正文

## 3.7 发送与审计

目标：

- 通过现有企微自建应用通道发送
- 记录成功、失败、重试和去重信息

交付物：

- delivery service
- digest audit log
- retry 机制

验收点：

- 发送失败不影响其他用户
- 可查到某天某个时间槽的发送记录
- 能对失败用户补发

## 3.8 联调与验收

目标：

- 用试点用户跑完整链路
- 校验文案、时点、风险准确性

交付物：

- 联调脚本
- 联调检查单
- 试点反馈记录

验收点：

- 试点用户收到内容与角色匹配
- 项目延期、高优 Bug、测试准出风险可稳定触发

## 4. 推荐开发顺序

建议顺序如下：

1. 配置加载与校验
2. 调度入口
3. 接收人解析
4. 研发 / 产品基础摘要
5. 测试 / 管理摘要
6. 风险判定引擎
7. 审计与重试
8. 即时提醒

原因：

- 先把稳定的固定摘要跑通
- 再补更敏感的风险升级和额外提醒

## 5. 建议测试用例

### 5.1 配置类

- 合法配置可加载
- 非法角色报错
- 重复 userid 报错
- 空 roles 报错

### 5.2 汇总类

- 研发用户能拿到我的任务和我的 Bug
- 产品用户能拿到我的需求
- 管理用户能按项目范围拿到进度摘要

### 5.3 风险类

- 超期任务进入风险列表
- blocked 且超期任务升级为 P0
- 高优 Bug 超期进入即时提醒候选
- 准出失败进入测试与管理摘要

### 5.4 发送类

- 单用户发送成功
- 单用户发送失败后可重试
- 同一时间槽重复执行不重复发送

## 6. 里程碑建议

### 里程碑 A：基础摘要可用

范围：

- morning / evening 两次调度
- 研发 / 产品摘要
- 发送与基础日志

### 里程碑 B：风险可用

范围：

- 项目延期
- 高优 Bug 超期
- 任务超期 / blocked
- 测试准出风险

### 里程碑 C：灰度上线

范围：

- 2 到 4 个试点用户
- 3 到 5 个工作日观察
- 调整阈值和文案

## 7. 角色分工建议

| 角色 | 建议负责内容 |
| --- | --- |
| 后端 / 脚本开发 | 调度、聚合、规则、发送、审计 |
| 产品 | 角色口径、文案、阈值确认 |
| 测试 | 场景覆盖、试点反馈、误报漏报核验 |
| 运维 / 管理员 | 配置维护、企微可见范围、定时任务观察 |

## 8. 风险点

- 禅道字段口径与实际页面不完全一致
- 用户 scope 配置不准会导致消息失焦
- 高优 Bug 阈值需要结合团队习惯调整
- 若真实环境数据量较大，批量查询需控制节奏

## 9. 开发完成定义

满足以下条件可视为“功能完成”：

- 工作日早晚摘要可稳定发送
- 四类角色都能得到正确视角
- 项目延期、Bug 超期、任务 blocked、准出失败可正确进入风险
- 多角色合并可正常工作
- 审计日志可追踪发送结果
