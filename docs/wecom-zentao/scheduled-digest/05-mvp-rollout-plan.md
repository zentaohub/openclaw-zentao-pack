# MVP 与上线分期

## 1. MVP 范围

第一版建议只做高价值、低复杂度部分：

- 工作日固定推送：09:00、18:00
- 角色：产品、研发、测试、管理
- 通道：企业微信自建应用私信
- 模板：简洁 markdown
- 内容：短摘要 + 风险提醒
- 风险范围：
  - 项目 / 执行延期
  - 任务延期
  - 高优 Bug 超期
  - 测试准出风险
  - 关闭准备度风险

## 2. MVP 明确不做

- 群聊推送
- 复杂卡片交互
- 用户自行配置推送时间
- 非工作日和节假日特殊规则
- 全量个性化字段开关

## 3. 推荐落地顺序

## 3.1 第一阶段：打通最小闭环

- 固定两次调度
- 读取用户角色与映射配置
- 产出研发 / 产品 / 测试 / 管理四类短消息
- 发私信
- 写发送审计日志

验收目标：

- 指定用户能稳定在 09:00、18:00 收到一条消息
- 消息内容不超过 8 行
- 同一用户不会重复收到多条同类消息

## 3.2 第二阶段：补齐风险策略

- 增加延期与阻塞判定
- 增加高优 Bug 超期阈值
- 增加测试准出与关闭准备度规则
- 增加重大风险即时提醒

验收目标：

- 能正确识别延期项目、超期 Bug、blocked 任务
- 风险优先级正确
- 即时提醒不会反复轰炸

## 3.3 第三阶段：优化使用体验

- 支持 template card
- 支持更多快捷入口
- 支持个人订阅开关
- 支持更多角色或按部门配置

## 4. 上线建议

建议按灰度方式推进：

1. 先给 2 到 4 个试点用户
2. 观察 3 到 5 个工作日
3. 根据反馈调整文案长度和风险阈值
4. 再扩大到完整角色群体

## 5. 验收标准

## 5.1 内容质量

- 消息简洁
- 风险项清楚
- 入口可用
- 无明显重复内容

## 5.2 业务准确性

- 任务超期判定准确
- 高优 Bug 超期判定准确
- 项目 / 执行延期判定准确
- 测试准出风险判定准确

## 5.3 工程可靠性

- 单用户失败不影响全局
- 支持失败重试
- 支持按日期和时间槽查询发送记录
- 支持手动补发

## 6. 后续扩展方向

- 节假日与调休日历接入
- 角色模板自定义
- 多语言模板
- 部门 / 标签群发摘要
- 管理驾驶舱周报
- 周一周报、周五周总结

## 7. 风险与注意事项

上线时要特别注意：

- 用户映射不完整时不要静默失败
- 角色范围配置过大时要避免消息失焦
- 风险阈值过严会造成打扰过多
- 风险阈值过松会失去提醒价值

建议首期以“宁简不乱”为优先原则。
