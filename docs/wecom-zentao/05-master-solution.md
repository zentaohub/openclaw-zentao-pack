# 企微机器人/自建应用对话式操作禅道完整方案

## 1. 项目背景与目标

当前云服务器 `1.14.73.166` 上已存在 `openclaw-zentao-pack`，它已经不是一个试验性脚本集合，而是一套面向 OpenClaw 主会话和企业微信机器人的禅道全流程技能包。已有能力覆盖查询、创建、状态流转、关联操作，以及企业微信消息回调、通讯录同步、身份匹配等关键链路。

本方案的目标不是从零重造一套系统，而是在既有能力基础上，形成一套可评审、可落地、可实施的“企微机器人/自建应用对话式操作禅道”完整方案。核心目标如下：

- 在企业微信中完成高频禅道操作，降低切换成本。
- 复用现有 Prompt、脚本、路由和总结文档，缩短落地时间。
- 同时支持企微机器人和企微自建应用两种接入形态。
- 把身份映射、权限校验、危险操作确认、审计追踪、失败恢复做成标准能力。

## 2. 现有基础能力盘点

根据仓库现有总结文件，服务器上的 `openclaw-zentao-pack` 已具备以下基础：

### 2.1 现有能力

- 查询：我的任务、产品、模块、项目、执行、需求、Bug、测试单、发布、验收、上线检查、关闭准备等。
- 创建：产品、模块、需求、任务、测试用例、测试单、Bug、发布。
- 流转：任务、需求、Bug、发布、测试单状态更新。
- 联动：发布关联、执行关联、测试单关联、团队成员添加。
- 企微能力：消息回调、我的任务回复、通讯录同步、企微与禅道身份匹配。

### 2.2 现有技术资产

- 主 Prompt/Agent：`agents/openai.yaml`
- Prompt 治理：`agents/modules/robot-prompt-governance.yaml`
- 快速路由建议：`openai-fast-routing-rules.md`
- 高频命令表：`禅道机器人高频指令路由表.md`
- 机器可读路由：`intent-routing.yaml`
- 禅道客户端：`scripts/shared/zentao_client.ts`
- 企微客户端：`scripts/shared/wecom_client.ts`
- 回调入口：`scripts/callbacks/wecom_callback.ts`
- 回复拼装：`scripts/replies/wecom_task_reply.ts`

这些资产意味着：本项目已经具备业务内核和执行骨架，缺的是产品级方案统一、工程化约束、交互规范与实施拆解。

## 3. 用户角色与权限模型

### 3.1 用户角色

- 研发：查任务、更新任务、查 Bug、处理 Bug。
- 测试：查测试单、执行用例、提 Bug、看准出。
- 产品：提需求、看需求、评审需求。
- 项目经理/交付：看执行、看发布、查验收、查关闭准备度。
- 负责人：审批或确认高风险动作。
- 管理员：处理映射、权限、路由和审计配置。

### 3.2 权限模型

权限不能只靠模型判断，必须有服务端确定性校验，建议分为四层：

- L1 模型层：识别意图与槽位，不做最终授权。
- L2 路由层：判断是否高风险、是否缺参数。
- L3 服务层：检查身份映射、组织权限、操作资格。
- L4 禅道层：以禅道真实权限作为最终执行依据。

### 3.3 身份映射原则

- 企微 `userid` 是统一入口身份。
- 禅道账号通过显式绑定表、邮箱或工号自动匹配。
- 自动匹配失败时进入待绑定状态。
- 未绑定身份时只允许查看帮助，不允许执行敏感动作。

## 5. MVP 范围与路线图

### 5.1 MVP

- 企微机器人优先接入。
- 打通身份映射、回调解析、快速路由、查询、创建、状态流转。
- 建立危险操作确认和审计闭环。
- 自建应用先提供跳转承接能力，不一开始做完整复杂前端。

### 5.2 路线图

- Phase 1：回调与身份打底、统一路由、核心查询与执行。
- Phase 2：多轮补问、候选选择、管理视图、异常恢复优化。
- Phase 3：自建应用工作台、复杂表单、任务中心和主动提醒。

## 6. 会话流程设计

### 6.1 核心处理流程

1. 接收企业微信消息或应用请求。
2. 解析发送者身份、会话类型、上下文。
3. 先按 `intent-routing.yaml` 与高频规则做快速匹配。
4. 命中后提取参数并补齐上下文。
5. 缺参数则补问，存在歧义则候选选择。
6. 危险动作则进入确认态。
7. 服务端执行授权与禅道调用。
8. 返回结构化结果并写审计日志。

### 6.2 高速路由原则

- 对 `我的任务`、`查项目`、`查执行`、`测试准出` 等高频命令直接短路执行。
- 参数齐全时直接执行，不展开长分析。
- 参数不足时只追问最关键缺口。
- 未命中高频命令时再进入通用推理。

### 6.3 多轮补问原则

- 一次只问一个关键缺口。
- 候选对象用序号返回，方便移动端选择。
- 使用 `继续` 承接最近未完成流程。
- 使用 `取消` 退出当前待确认或待补问流程。

## 7. 核心场景细化

### 7.1 我的任务

- 用户输入：`我的任务`
- 系统自动完成：企微身份识别 -> 禅道身份映射 -> 查询我的任务 -> 返回列表卡片
- 风险：身份映射缺失
- 兜底：提示绑定账号

### 7.2 创建任务

- 用户输入：`在执行 4 建个任务，标题是补齐企微映射`
- 系统识别：`create-task`
- 必要字段：`execution`、`name`
- 执行后回执：任务 ID、当前状态、负责人、后续动作

### 7.3 完成任务

- 用户输入：`完成任务 33`
- 系统识别：`update-task-status`
- 流程：校验状态 -> 校验权限 -> 返回确认卡 -> 执行 -> 返回回执

### 7.4 测试准出

- 用户输入：`测试准出 4`
- 默认理解：`execution=4`，若上下文有测试单则优先 `testtask`
- 返回内容：准出结论、阻塞项、建议动作

### 7.5 创建 Bug

- 用户输入：`提个 Bug`
- 若缺 `product`、`title`、`builds`，则逐步补问
- 机器人模式完成最小创建；复杂复现步骤可跳自建应用补齐

## 8. 企业微信消息 UI 与卡片设计

### 8.1 输出骨架

建议保留现有结构化语义：

- `【角色】`
- `【意图】`
- `【结果】`
- `【正文】`
- `【后续动作】`

在纯文本场景中直接使用该结构；在卡片场景中则把这五部分映射为标题、状态、正文区、操作区和引导区。

### 8.2 卡片类型

- 列表卡片：用于任务、Bug、执行、测试单、发布列表。
- 详情卡片：用于单对象详情展示。
- 确认卡片：用于高风险动作前确认。
- 候选卡片：用于多匹配对象选择。
- 失败卡片：用于权限不足、超时、接口失败。
- 跳转卡片：用于进入自建应用补齐复杂字段。

### 8.3 交互策略

- 列表卡片最多给 2 到 3 个主动作。
- 危险动作不直接挂在列表页上，而是先进入确认卡片。
- 移动端优先短文案、少字段、数字化选择。

## 9. 系统架构与模块设计

### 9.1 架构分层

- 消息接入层：回调、验签、解密、去重。
- 会话编排层：路由、槽位提取、上下文、确认态。
- 身份与权限层：企微-禅道映射、组织角色、授权。
- 禅道适配层：统一 API 封装、错误分类、对象操作。
- 审计与观测层：日志、追踪、告警、审计查询。

### 9.2 模块复用建议

- 继续复用 `scripts/shared/zentao_client.ts` 作为禅道客户端核心。
- 继续复用 `scripts/shared/wecom_client.ts` 处理企微访问和通讯录同步。
- 保留 `intent-routing.yaml` 作为高频路由基础。
- 以 `agents/openai.yaml` 承接复杂语义推理和输出治理。

## 10. 主动通知与变更触达方案

在当前“企微消息回调 -> 禅道执行”的基础上，建议补齐“禅道变更 -> 企微自建应用主动通知”能力，作为 Phase 2 的默认增强项。

### 10.1 目标

- 让关键变更自动触达对应责任人
- 让 PM 只接收与产品判断相关的变更
- 让测试、开发、项目负责人收到下一步需要动作的消息
- 避免所有状态变化都打扰所有人

### 10.2 第一版推荐覆盖场景

- 需求：新增、核心字段变更、关闭/拒绝、优先级变化、计划变化、完成/验收
- Bug：高优新建、高优重开、高优修复、高优关闭、优先级升级、影响需求验收
- 任务：转派、关键任务完成、关键任务阻塞/延期、关联需求任务关闭

### 10.3 第一版推荐落地方式

- 优先在现有动作脚本成功后直接触发通知
- 例如：
  - `update_task_status.ts`
  - `update_bug_status.ts`
  - `assign_bug.ts`
- 先覆盖“通过 OpenClaw / 当前脚本执行的变更”
- 后续如果要覆盖“禅道后台人工直接改状态”，再补 webhook 或定时轮询 action 日志

### 10.4 配套文档

- `11-notification-rules-mvp.yaml`
  - 定义命中什么事件、通知谁、PM 是 direct / cc / none
- `12-notification-templates-mvp.yaml`
  - 定义企微 markdown 消息模板
- `13-receiver-resolution-spec.md`
  - 定义接收人如何从禅道对象字段中解析

## 11. 数据流与集成关系

### 11.1 查询流

企微消息 -> 回调服务 -> 路由层 -> 身份映射 -> 禅道查询 -> 卡片/文本回执 -> 审计日志

### 11.2 执行流

企微消息 -> 路由层 -> 参数补齐 -> 权限判断 -> 确认卡 -> 用户确认 -> 禅道执行 -> 回执 -> 审计日志

### 11.3 通讯录同步流

### 11.4 主动通知流

业务脚本执行成功 -> 产出标准事件 -> 通知规则匹配 -> 接收人解析 -> 企微自建应用发送消息 -> 写通知日志

企微通讯录回调 -> 用户同步服务 -> 更新映射候选信息 -> 触发绑定 / 刷新缓存

## 12. 异常处理与控制

### 12.1 对象不存在

- 返回“未找到对象”，建议用户改用 ID 或重新查询。

### 12.2 权限不足

- 区分身份未绑定、角色不足、禅道权限不足三类反馈。

### 12.3 超时与失败

### 12.4 通知失败

- 业务主流程成功后，通知应视为异步附加动作，避免因为企微发送失败回滚业务状态。
- 记录失败日志，支持后续重试。
- 接收人解析失败时，要明确记录是哪个角色未能解析，不要静默吞掉。

- 查询允许直接重试。
- 执行动作要区分未执行成功和结果未知。
