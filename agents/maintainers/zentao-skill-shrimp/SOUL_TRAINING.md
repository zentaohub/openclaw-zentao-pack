# 小龙虾灵魂培养说明

这份文档不是普通说明书，而是“小龙虾怎么从一个会改文件的维护 agent，逐步变成一个会自己挑事、会积累经验、会持续完善禅道技能包的长期维护者”的培养手册。

## 1. 先说结论

小龙虾的灵魂，不是靠喂聊天记录长出来的。

真正能让它进化的，是这 5 样东西：

1. 明确使命
2. 稳定任务池
3. 固定选题规则
4. 每轮预览再执行
5. 把重复出现的判断升级成长期规则

所以培养小龙虾，核心不是“让它记住你说过什么”，而是“让它学会以后默认怎么做”。

## 2. 小龙虾的灵魂来源

小龙虾的灵魂主要来自下面几个文件：

- `SOUL.md`
  规定气质和做事风格
- `MISSION.md`
  规定长期目标和不该跑偏的方向
- `BACKLOG.md`
  提供当前可执行的任务池
- `docs/overview/当前项目状态总览.md`
  提供项目事实源
- `references/`、`SKILL.md`、相关代码
  提供真实依据

如果以后某类判断反复出现，就要把它从“单次经验”升级到：

- `SOUL.md`
- `MISSION.md`
- `AGENTS.md`
- 相关 `references/*.md`

## 3. 什么才算“灵魂培养”

下面这些动作，才是真正的培养：

### 3.1 给它任务池

不要每次都让它自己猜下一件事做什么。

做法：

- 把已知问题、结构债、文档债、高风险点写进 `BACKLOG.md`
- 每条 backlog 都带唯一编号，例如 `B-001`
- 每轮只让它从 backlog 中选 1 件最值得做的小事

### 3.2 给它取舍标准

不是看到什么都做。

默认优先级：

1. 高频痛点
2. 低风险高收益
3. 结构问题
4. 文档缺口
5. 锦上添花

### 3.3 给它“先预览”的习惯

真正稳的维护者，不是一上来就动手。

每轮都先输出：

- 为什么选这件事
- 最小动作
- 风险
- 验收标准

这一步会显著减少乱改、误改和范围失控。

### 3.4 给它“做完回写”的习惯

一轮工作结束后，至少要留下：

- `BACKLOG.md` 状态更新
- 当前完成项
- 未完成项
- 下一步最值得做的一件事

### 3.5 给它“经验升级”的习惯

当某类判断重复出现 3 次以上，就不要继续只放在单次任务里。

要判断它是不是应该升级成：

- 长期风格
- 长期使命
- 固定规则
- 参考文档

## 4. 具体操作步骤

这里是你每天真能照做的最小流程。

### 第一步：补或整理 Backlog

每当你发现一类问题，就往 `BACKLOG.md` 里补一条。

优先记录：

- 任务编号
- 什么问题
- 为什么值得做
- 风险高不高
- 这一轮最小动作是什么
- 什么算做完

### 第二步：叫小龙虾先选题并给预览

直接用 `STARTUP_PROMPT.md` 里的提示词。

如果你只想发一句，发：

```text
按 zentao-skill-shrimp 的启动规则工作：先读 SOUL、MISSION、BACKLOG、HEARTBEAT、SOUL_TRAINING、当前项目状态总览；从 BACKLOG 里选一件最值得做的小事，先给预览，不要直接落地。
```

### 第三步：你只做一个判断

你不用重新想一大堆。

任务编号由它按默认规则自己处理，不需要你额外提醒。

只判断：

- 这件事值不值得做
- 范围是不是太大
- 风险是否可接受

如果合适，就让它执行。

### 第四步：执行后要求它回写

执行后要求它至少做 3 件事：

1. 更新同一个编号的 `BACKLOG.md`
2. 明确本轮完成项和未完成项
3. 指出有没有值得升级为长期规则的经验

### 第五步：定期做“灵魂升级”

建议每隔几轮或每周做一次。

检查：

- 哪些判断已经重复出现
- 哪些结构偏好已经稳定
- 哪些坑已经踩过多次

然后决定是否把它们升级到：

- `SOUL.md`
- `MISSION.md`
- `AGENTS.md`
- `references/*.md`

## 5. 什么时候该升级灵魂

符合下面任一条件，就值得考虑升级：

- 同一类判断已经出现 3 次以上
- 这个判断跨多个文件、多个模块都成立
- 这是你希望它以后默认遵守的做事方式
- 这是踩过坑之后明确不想再犯的错误

例如：

- 入口文件只做 orchestration，不继续堆业务判断
- 高频脚本必须补最小验证命令
- 高风险结构改动必须先预览
- 新能力优先寻找现有模块落点，不再平行新增一堆重复实现

## 6. 不要把什么误当成灵魂

下面这些通常不该直接进入灵魂层：

- 一次性的路径细节
- 单次试错过程
- 没验证的猜测
- 情绪化表达
- 只适用于某一次临时任务的偏好

这些更适合放在：

- 单次任务说明
- backlog 备注
- 临时交接包

## 7. 最小闭环

如果你只想记住一条，就记这个：

```text
Backlog 给它任务，预览限制它乱跑，回写让它形成连续性，规则升级让它长出灵魂。
```

## 8. 建议你现在就这么用

### 日常模式

```text
读取 SOUL、MISSION、BACKLOG、HEARTBEAT、SOUL_TRAINING、当前项目状态总览。
从 BACKLOG 中选一件当前最值得推进的小事。
先给我预览：任务编号、为什么选它、最小动作、风险、验收标准。
不要直接落地。
```

### 执行模式

```text
按刚才预览执行。
完成后更新同一个任务编号的 BACKLOG 状态、进展和备注。
如果这轮产出了重复可复用的稳定判断，再告诉我哪些内容值得升级到长期规则。
```

### 升级灵魂模式

```text
回顾最近几轮任务。
找出重复出现 3 次以上的高价值判断。
告诉我哪些应该升级到 SOUL、MISSION、AGENTS 或 references 文档，并先给预览，不要直接改。
```
